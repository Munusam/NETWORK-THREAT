## рҹҸ—пёҸ Architecture Overview

This project implements a **complete Big Data lifecycle** across **8 distinct layers**, from raw traffic ingestion all the way to real-time monitoring.

<br/>

| # | Layer | Technology | Role |
|---|-------|------------|------|
| 1пёҸвғЈ | **Ingestion** | Apache Kafka | Real-time network log generation |
| 2пёҸвғЈ | **Storage** | MinIO (S3-compatible) | Delta Lake / Lakehouse architecture |
| 3пёҸвғЈ | **Processing** | Spark Structured Streaming | Real-time data transformation |
| 4пёҸвғЈ | **Orchestration** | Apache Airflow | Automated pipeline scheduling |
| 5пёҸвғЈ | **Machine Learning** | Spark MLlib | Random Forest threat classification |
| 6пёҸвғЈ | **Tracking** | MLflow | Experiment tracking & model versioning |
| 7пёҸвғЈ | **Serving** | BentoML | Containerized REST API deployment |
| 8пёҸвғЈ | **Monitoring** | Prometheus + Grafana | Real-time system & inference tracking |

```bash
git clone https://github.com/Munusam/NETWORK-THREAT.git
cd NETWORK-THREAT
```

### 2. Launch the Infrastructure

Start all core services вҖ” Kafka, MinIO, MLflow, Spark, Airflow, BentoML, Prometheus, and Grafana вҖ” in detached mode:

## рҹ—әпёҸ End-to-End Execution Guide

Work through these steps to push data through every layer of the pipeline:

---

### рҹҹў Step 1 вҖ” Start the Kafka Generator *(Layer 1)*

Generate a continuous stream of simulated benign and malicious network traffic:

```bash
python generator.py
```

---

### рҹҹЎ Step 2 вҖ” Stream Data to the Lake *(Layers 2 & 3)*

1. Open **JupyterLab** вҶ’ [http://localhost:8888](http://localhost:8888)
2. Open the `01_data_ingestion.ipynb` notebook.
3. Run all cells to launch Spark Structured Streaming вҖ” this consumes the Kafka stream and writes Parquet files into the **MinIO Lakehouse** ([http://localhost:9001](http://localhost:9001)).

---

### рҹ”ө Step 3 вҖ” Train & Track the Model *(Layers 5 & 6)*

1. Open `02_model_training.ipynb` in JupyterLab.
2. Run all cells to train the **Random Forest classifier** on the Delta Lake data.
3. Visit **MLflow UI** вҶ’ [http://localhost:5050](http://localhost:5050) to inspect F1 scores, run metrics, and registered model versions.
4. *(Optional)* Copy the `Run ID` of your best model to update the BentoML container when deploying a new version.

---

### рҹҹЈ Step 4 вҖ” Serve the Model *(Layer 7)*

The BentoML API starts automatically. To test it:

1. Navigate to **Swagger UI** вҶ’ [http://localhost:3030](http://localhost:3030)
2. Open the `POST /predict` endpoint вҶ’ click **Try it out** вҶ’ send a payload:

```json
{
  "source_port": 12345,
  "dest_port": 80,
  "bytes": 14000
}
```

3. Receive an instant classification:

```json
{ "is_attack": 1 }   // вҡ пёҸ  Malicious traffic detected
{ "is_attack": 0 }   // вң…  Benign traffic
```

---

### рҹҹ  Step 5 вҖ” Monitor the API *(Layer 8)*

1. Open **Grafana** вҶ’ [http://localhost:3050](http://localhost:3050)  
   *(Login: `admin` / `admin`)*
2. View the custom dashboard tracking:

| Metric | PromQL Query |
|--------|--------------|
| **Total API Requests** | `sum(bentoml_service_request_duration_seconds_count)` |
| **Model Inference Speed** | `sum(bentoml_service_request_duration_seconds_sum)` |
| **Active Traffic** | `sum(bentoml_service_request_in_progress)` |

---

### рҹ”ҙ Step 6 вҖ” Orchestrate Background Jobs *(Layer 4)*

1. Open **Apache Airflow** вҶ’ [http://localhost:8080](http://localhost:8080)  
   *(Login: `admin` / `admin`)*
2. Unpause and manually trigger the `network_threat_orchestration` DAG.
3. Watch the automated tasks run **Model Drift Monitoring** and **Cold Data Archiving**.
